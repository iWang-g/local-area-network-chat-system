# 服务端（vs-server）跨平台实现说明

本文说明当前 `vs-server` 与 Windows 的绑定点，以及为支持 **Windows 与 Linux（如 Ubuntu）** 并行的推荐改造路径。内容按仓库内真实源文件与 API 使用关系组织。

---

## 1. 当前状态

- 工程为 **Visual Studio**（`vs-server.vcxproj` / `vs-server.sln`），无统一 CMake。
- 源码中大量使用 **Win32 / Winsock / WinHTTP / BCrypt / Crypt32**，以及 **运行时加载 `sqlite3.dll`**。
- 因此 **不能** 在 Linux 上零改动直接编译；需移植后再于 Ubuntu 等平台编译，生成 ELF 可执行文件。

---

## 2. 总体思路：一套业务源码 + 薄适配层

- 协议编解码、`database` 内 SQL 与业务、`wire` 等与平台无关的逻辑尽量保持语义不变。
- 将 **系统调用** 收敛到少数模块或通过 `#ifdef` / 独立 `platform_*.cpp` 实现。
- **构建**：建议引入 **CMake**，在 Windows 上生成 MSVC 工程、在 Linux 上生成 Makefile/Ninja，统一源文件列表与宏定义。
- **网络**：在 **Winsock** 与 **POSIX socket** 之上做薄封装（类型、关闭、错误码、是否需 `WSAStartup`）。
- **SQLite**：Linux 上推荐 **链接 `libsqlite3`**（如 `libsqlite3-dev`），比继续动态加载更易维护；Windows 可继续动态加载或改为链接导入库，与 Linux 对齐。
- **HTTP（mail-helper）**：用 **libcurl** 替代 WinHTTP，两平台均可使用。
- **Base64 与安全随机数**：使用不依赖 `crypt32` / `bcrypt` 的实现（自写 RFC4648、或 OpenSSL 等），避免平台差异。

---

## 3. 按源文件拆解

### 3.1 `src/net/tcp_server.cpp`：Winsock 与类型差异

**现状要点：**

- 包含 `WinSock2.h`、`WS2tcpip.h`。
- 入口使用 **`WSAStartup` / `WSACleanup`**（Linux 不需要）。
- 使用 **`SOCKET`、`INVALID_SOCKET`、`SOCKET_ERROR`、`closesocket`**。
- `setsockopt(..., SO_REUSEADDR, ...)` 使用 **`BOOL opt`**（MSVC）；在 Linux 上应为 **`int opt = 1`**，否则可能无法通过编译或行为异常。

**跨平台建议：**

- 用统一类型别名，例如：
  - Windows：`typedef SOCKET socket_t`，使用 `INVALID_SOCKET`、`closesocket`。
  - Linux：`typedef int socket_t`，无效为 `-1`，使用 `::close` 关闭套接字。
- 仅在 `_WIN32` 下调用 `WSAStartup` / `WSACleanup`；Linux 分支省略。
- `bind` / `listen` / `accept` / `send` / `recv` 主流程可大量保留；错误处理可封装：Linux 用 `errno`，Windows 用 `WSAGetLastError()`。

`std::thread(handleClient, ...)` 与平台无关，可保留。

---

### 3.2 `src/net/mail_helper_http.cpp`：WinHTTP

**现状要点：**

- `_dupenv_s`（MSVC）、`MultiByteToWideChar`、**`WinHttpOpen` / `Connect` / `OpenRequest` / `SendRequest` / `ReceiveResponse` / `QueryHeaders`** 等。

**跨平台建议：**

- 使用 **libcurl** 发 **HTTPS POST**，请求体仍为现有 JSON。
- 环境变量 **`LANCS_MAIL_HELPER_URL`**、**`LANCS_MAIL_HELPER_SECRET`** 使用标准 **`std::getenv`**（Windows/Linux 一致）。
- URL 由 curl 处理，可不再手写 `WinHttpCrackUrl`。
- CMake 中 `find_package(CURL)` 或固定链接 `libcurl`；注意 TLS/CA 与 WinHTTP 行为需联调。

未设置 `LANCS_MAIL_HELPER_URL` 时行为应与现有 `mailHelperConfigured()` 一致（不发外网请求）。

---

### 3.3 `src/utils/password_hash.cpp`：BCrypt 随机数

**现状要点：**

- `BCryptGenRandom`（`bcrypt.lib`）用于随机盐、随机 token、六位验证码等。

**说明：**

- 密码存储链为 **picosha2 迭代 SHA256**（`derivePasswordHash`），与平台无关。

**跨平台建议：**

- 封装 `fillRandomBytes(unsigned char *buf, size_t n)`：
  - Linux：**`getrandom(2)`** 或 **`/dev/urandom`**；
  - Windows：可继续 **`BCryptGenRandom`**，或两端统一为 **OpenSSL `RAND_bytes`**。

---

### 3.4 `src/net/message_parse.cpp`、`src/net/file_relay.cpp`：Crypt32 Base64

**现状要点：**

- `wireBase64Encode` / `wireBase64Decode` 等使用 **`CryptBinaryToStringA` / `CryptStringToBinaryA`**（`crypt32.lib`）。
- `file_relay.cpp` 中另有 **`CryptStringToBinaryA`** 用于长度探测等。

**跨平台建议：**

- 实现或引入 **RFC4648** 的 Base64 编解码（可抽成 `base64.hpp` / `base64.cpp`），**`message_parse` 与 `file_relay` 共用**，删除对 `crypt32` 的依赖。

`file_relay` 中 **`std::filesystem`** 路径逻辑与 Linux 兼容；主要还需替换 Base64 与下文「可执行目录」。

---

### 3.5 `src/db/database.cpp`：可执行目录与创建目录

**现状要点：**

- `GetModuleFileNameW` 得到 exe 目录，`exeDir + L"\\data"`，**`CreateDirectoryW`**。

**跨平台建议：**

- 抽象 **`std::filesystem::path appExecutableDir()`**：
  - **Windows**：保留 `GetModuleFileNameW` + 取父路径；
  - **Linux**：解析 **`/proc/self/exe`**（`readlink` + `canonical`），失败则回退 **`std::filesystem::current_path()`**。
- 使用 **`exeDir / "data"`**（`std::filesystem::path`），避免手写反斜杠。
- 创建目录使用 **`std::filesystem::create_directories`**（与 `logger` 已用方式一致）。

宽字符与 `WideCharToMultiByte` 可收缩为 Windows 专用；Linux 统一 UTF-8 字符串 + `path`。

---

### 3.6 `src/db/sqlite_dynamic.cpp`：加载 `sqlite3.dll`

**现状要点：**

- **`LoadLibraryW` / `GetProcAddress`** 加载 **`sqlite3.dll`**。

**跨平台建议（推荐简化）：**

- **Linux**：编译时链接 **`libsqlite3`**，直接调用 `sqlite3_*`，可将 `SqliteDynamic` 改为对静态符号的薄包装，或去掉动态加载、全局 `#include <sqlite3.h>`。
- **Windows**：可选用 `.dll` 动态加载，或同样改为链接 `.lib`，减少 `#ifdef` 分支。

若坚持运行时加载：Linux 使用 **`dlopen("libsqlite3.so")`** + **`dlsym`**，并在文档中说明 **`LD_LIBRARY_PATH`** 或 rpath。

---

### 3.7 `src/net/file_relay.cpp`：`exeDirectoryW` 与 Crypt32

**现状要点：**

- 与 `database.cpp` 类似的 **`GetModuleFileNameW`** 构造 **`data/offline_partial`、`data/sticker_cache`** 等路径。
- Base64 依赖 **`CryptStringToBinaryA`**。

**跨平台建议：**

- 与数据库模块 **共用** `appExecutableDir()`；路径统一用 `std::filesystem::path`。
- Base64 与 **`message_parse`** 共用实现。

---

### 3.8 `src/utils/logger.cpp`

**现状要点：**

- 已对 **`localtime_s` / `localtime_r`** 做 `_WIN32` 分支；控制台 UTF-8 仅 Windows。

**跨平台建议：**

- Linux 分支已基本就绪；可选增加 Linux 下 locale/终端 UTF-8 说明，非必须。

---

## 4. CMake 与依赖（示例）

| 平台 | 典型依赖 |
|------|----------|
| Ubuntu | `build-essential`、`cmake`、`libsqlite3-dev`、`libcurl4-openssl-dev`（或等价包名） |

建议落地顺序：

1. 添加 **CMake**，Windows/Linux 共用同一源列表。
2. 封装 **TCP 层**（WSA、socket 类型、`setsockopt` 参数类型）。
3. 替换 **Base64**（去掉 `crypt32`）。
4. 替换 **随机数**（去掉 `bcrypt` 或统一 OpenSSL）。
5. **`mail_helper_http` → libcurl**。
6. **路径 + SQLite 链接**（`appExecutableDir` + `-lsqlite3`）。
7. 在 Ubuntu 上验证：`./vs-server <port>`、`data/chat.db`、`data/offline_partial`、`logs/`。

---

## 5. 风险与一致性

- **协议与字节序**：若 `wire` 已按网络字节序处理，跨平台通常无差异；重点在 socket 错误码与边界条件。
- **HTTPS**：curl 在 Linux 上使用系统 CA；与 WinHTTP 校验链差异需联调。
- **数据目录**：保持「相对可执行文件目录的 `data/`」规则一致，便于备份与迁移。

---

## 6. 小结

当前耦合集中在：**Winsock、WinHTTP、BCrypt、Crypt32、可执行路径（宽字符 API）、`sqlite3.dll` 动态加载**。跨平台方向是：**POSIX/BSD socket + libcurl + 统一随机数/Base64 + `std::filesystem` + 链接 libsqlite3**，并用 **CMake** 统一 Windows 与 Linux 的构建入口。
