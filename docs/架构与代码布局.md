# 架构与代码布局

## 1. 总体架构（与论文一致）

- **Qt 客户端**：仅使用 **Qt Widgets**（`.ui` + C++），通过 **`QTcpSocket`** 与聊天主服务端维持 **TCP 长连接**；不直连 SMTP。
- **聊天主服务端（vs-server）**：标准 **C++**，**Socket** 接入与协议处理，**SQLite** 持久化；可部署在 **Windows / Linux**。
- **邮件辅助服务（可选、建议独立进程）**：负责 **SMTP 发信**；主服务端通过**内网 HTTP**（或同类轻量接口）调用，并做**密钥/令牌**校验。实现技术栈（如 Node）以实现阶段选型为准。

客户端只连主服务端；验证码链路：**客户端 → 主服务端 →（必要时）邮件服务 → 用户邮箱**。

## 2. 仓库目录（当前约定）

```
local-area-network-chat-system/
├── qt-client/          # Qt Creator 打开 CMakeLists.txt，独立构建
│   ├── src/
│   │   ├── app/        # main.cpp
│   │   ├── ui/         # 窗体（.ui / QWidget 子类）
│   │   ├── style/      # 全局 QSS、样式封装
│   │   └── utils/      # 工具函数
│   └── resources/      # 图片、.qrc 等
├── vs-server/          # Visual Studio 打开 vs-server.sln，独立构建
│   ├── src/            # 源码（含 main）；可扩展 net/、db/ 等子目录
│   └── include/vsserver/   # 对外头文件（如协议常量）
└── docs/               # 本目录
```

**不要求**仓库根目录提供“一键编两端”的统一 CMake；两端版本与发布各自维护即可。

## 3. 客户端分层（实现时遵守）

| 层级 | 职责 | 典型位置 |
|------|------|-----------|
| UI | 界面与交互，不直接拼二进制协议 | `src/ui/`、`src/style/` |
| 业务 | 登录态、会话列表、消息模型、校验规则 | 可在 `src/` 下增 `model/` 或 `core/`（按需） |
| 网络 | `QTcpSocket`、粘包/半包、报文组包/解析 | 可在 `src/` 下增 `net/`（按需） |

## 4. 服务端模块（实现时遵守）

| 模块 | 职责 | 建议位置 |
|------|------|-----------|
| 接入 | 监听、accept、连接生命周期、心跳 | `src/net/` 或等价 |
| 协议 | 解析、鉴权、路由、错误码 | 与接入同目录或 `src/protocol/` |
| 数据 | SQLite 访问、迁移、事务 | `src/db/` 或等价 |
| 邮件回调 | 调用辅助服务、限流与审计 | 与业务调度放一起，避免散落在各处 |

## 5. 非功能目标（开发自测参考）

- 局域网条件下文字消息交互尽量在**百毫秒量级**（理想环境，非硬性 SLA）。  
- 服务端支持**实验室规模**并发（例如数十路连接）稳定运行。  
- 密码**加盐哈希**入库；验证码**有效期、次数、频率**限制；辅助服务接口**不可被公网随意调用**。
