# 协议与接口草案

> 状态：**草案**。实现时可细化字段与枚举，但应保持**版本号/魔数**可识别，便于联调与论文描述一致。

## 1. 主链路：客户端 ↔ 聊天服务端（TCP）

### 1.0 已实现（阶段 1 联调）

- **成帧**：**4 字节大端无符号长度**（仅表示后续 JSON 的字节数，不含自身）+ **UTF-8 JSON** 文本（紧凑格式即可）。单帧负载上限 **256 KiB**。  
- **握手**：客户端发送 `{"type":"hello","magic":"LNCS","version":1}`；服务端校验通过后回复 `{"type":"hello_ok","version":1}`，否则 `{"type":"error","code":1001,"message":"bad handshake"}`。  
- **心跳**：客户端发送 `{"type":"heartbeat"}`；服务端回复 `{"type":"heartbeat_ack"}`。  
- **代码位置**：服务端 `vs-server/src/net/`（`wire`、`message_parse`、`tcp_server`）；客户端 `qt-client/src/net/lan_tcp_client.*`。默认端口 **`28888`**（`vsserver::kDefaultTcpPort`）。  
- **服务端当前实现**：**WinSock 2**（`ws2_32.lib`），便于 Windows + VS 联调；Linux 需在后续迭代替换为可移植网络层。

### 1.0.1 已实现（阶段 2：认证与 SQLite）

- **依赖**：Windows 服务端在**运行时**加载 **`sqlite3.dll`**（与 `vs-server.exe` 同目录优先）。官方包：[sqlite-dll-win-x64](https://www.sqlite.org/download.html)（直链目录通常为 `https://www.sqlite.org/{发布年份}/sqlite-dll-win-x64-{版本号}.zip`，因站点下载页依赖 JS，脚本里写死 URL 更稳）。仓库内提供 **`vs-server/third_party/sqlite/sqlite3.dll`**（可通过 **`vs-server/scripts/fetch-sqlite-dll.ps1`** 重新下载）；**x64** 工程生成后会自动将该 DLL **复制到 `$(OutDir)`**。不嵌入 `sqlite3.c` 进可执行文件。  
- **库文件**：`vs-server/src/db/`（`sqlite_dynamic` 动态加载、`database` 建表与 DAO）、`vs-server/src/utils/password_hash.cpp`（盐与口令哈希）。数据库文件默认 **`exe同目录/data/chat.db`**，表结构对齐论文表 4-1 `users`。  
- **握手后业务**：未完成握手时除 `hello` 外下发 `{"type":"error","code":2010,"message":"请先完成握手"}`（心跳亦同）。  
- **登录**：C→S `{"type":"auth_login","email":"...","password":"..."}`；成功 S→C `{"type":"auth_ok","user_id":<int>,"token":"<hex>","email":"..."}`；失败 `error`，`code` 见下表。  
- **注册**：C→S `{"type":"auth_register","email":"...","password":"...","email_code":"...","nickname":"..."}`（`nickname` 可省略）。须与未过期且通过校验的注册验证码匹配。成功帧同 `auth_ok`。  
- **错误码（与认证相关）**：`1001` 握手失败；`2001`～`2005`、`2010` 见 `protocol.hpp`；`2006`/`2007`/`2008` 见 §1.0.2。  
- **客户端**：登录/注册均可配置 **服务器地址 / 端口**（`QSettings`）；登录流程 **TCP → `hello` → `auth_login` → `auth_ok`**；注册流程见 §1.0.2。  

### 1.0.2 已实现（阶段 3·邮件验证码骨架）

- **表**：`email_codes`（对齐论文表 4-2），字段含 `email`、`code`、`purpose`、`expires_at`、`created_at`、`attempts`。  
- **请求验证码**：C→S `{"type":"req_email_code","email":"...","purpose":"register"}`（`purpose` 可省略，默认 `register`）。成功 S→C `{"type":"email_code_ok"}`；失败 `error`。同一邮箱 **60s** 内重复请求返回 **`2007`**，可选字段 **`retry_after_sec`**（秒）供客户端倒计时。验证码 **5 分钟**有效；错码累计 **5 次** 作废当前验证码。  
- **mail-helper（Node.js）**：仓库根目录 **`mail-helper/`**，`POST /v1/send-register-code`，JSON 体 `email`、`code`、`purpose`（默认 `register`）；可选 `Authorization: Bearer <密钥>`（与辅助进程 `MAIL_HELPER_SECRET`、服务端 `LANCS_MAIL_HELPER_SECRET` 一致）。`SMTP_URL` 为空时辅助服务 **dry-run**（仅打日志、仍返回 200），便于联调。  
- **vs-server 侧**：若设置环境变量 **`LANCS_MAIL_HELPER_URL`**（完整 URL，如 `http://127.0.0.1:8787/v1/send-register-code`），`req_email_code` 落库后会 **WinHTTP POST** 通知 mail-helper；失败则 **回滚本条验证码** 并返回 **`2008`**（邮件发送失败）。未设置该变量时行为同前：**仅服务端日志**含明文验证码。  
- **错误码补充**：`2006` 验证码错误或已过期；`2007` 请求验证码过于频繁；`2008` 已配置邮件辅助 URL 但通知或发信链路失败。  
- **客户端**：注册页 **获取验证码**（带倒计时）、**验证码输入框**；先 `req_email_code` 再提交注册。

### 1.1 传输与成帧

- 连接：**TCP**；正文采用 **长度前缀 + UTF-8 JSON**（与 1.0 一致）。  
- **粘包 / 半包**：客户端 `LanTcpClient`、服务端 `FrameAssembler` 已按长度字段拆帧。  
- 握手已交换：`magic`=`LNCS`、`version`=1（与 `vsserver::protocol.hpp` 一致）。

### 1.2 报文公共字段（建议）

每条业务报文建议包含：

- `type`：消息类型（枚举）  
- `seq`：请求序号（客户端生成，用于匹配响应）  
- `payload`：业务体  

### 1.3 消息类型（初稿，可扩展）

| type（示例名） | 方向 | 说明 |
|----------------|------|------|
| `auth_login` | C→S | 登录（邮箱/账号 + 密码或 token） |
| `auth_register` | C→S | 注册（邮箱、验证码、密码） |
| `auth_refresh` | C→S | 会话续期（若采用 token） |
| `req_email_code` | C→S | 请求发验证码（触发服务端校验频率后调用邮件链路） |
| `friend_search` | C→S | 按邮箱/账号查用户 |
| `friend_request` | C→S / S→C | 发起申请 / 推送通知 |
| `friend_handle` | C→S | 同意/拒绝 |
| `friend_list` | C→S / S→C | 列表同步 |
| `msg_send` | C→S | 点对点文字消息 |
| `msg_push` | S→C | 下行推送（含离线后的补发策略） |
| `msg_ack` | 双向 | 已送达/已读（粒度以实现为准） |
| `file_offer` / `file_chunk` / `file_ack` | 双向 | 分片传输元数据与进度（对齐论文 4.2.4） |
| `hello` | C→S | 握手（已实现） |
| `hello_ok` | S→C | 握手成功（已实现） |
| `heartbeat` | C→S | 保活（已实现） |
| `heartbeat_ack` | S→C | 心跳响应（已实现） |
| `error` | S→C | 统一错误承载（握手失败等，已实现 code 示例） |

### 1.4 错误码（建议单独维护）

在代码中集中定义枚举或常量表，文档中记录：**码 → 含义 → 客户端提示文案**。避免魔法数散落在 `if` 里。

## 2. 主服务端 ↔ 邮件辅助服务（HTTP，内网）

### 2.1 安全

- **HTTPS 可选**；局域网演示可用 HTTP + **固定 API Key / HMAC** 放在 Header。  
- **限流**：按邮箱/IP/全局限频，与主服务端验证码策略一致。

### 2.2 接口草案（示例）

`POST /api/send-code`

```json
{
  "to": "user@example.com",
  "subject": "验证码",
  "text": "您的验证码是 123456，5 分钟内有效。"
}
```

响应：`200` + `{ "ok": true }`；失败返回 `4xx/5xx` + 可读错误信息（仅供服务端日志与排错，勿直接暴露给最终用户敏感细节）。

具体路径与字段以实现仓库中辅助服务为准；**本文档只固定“主服务端不直接 SMTP”的职责边界**。

## 3. 与头文件对齐

服务端仓库中 `include/vsserver/protocol.hpp` 可放置 **魔数、协议版本**；扩展时同步更新客户端侧等价定义（或后续抽共享头文件，非必须）。
