# 开发路线

按依赖关系分阶段推进；每阶段结束应有**可运行、可演示**的增量。顺序可根据课时微调，但不建议跳过“协议成帧 + 登录”直接去写复杂 UI。

**界面总方向**：客户端为**即时通讯场景**，整体**借鉴 QQ（腾讯 QQ）类产品的布局与视觉习惯**（左侧功能栏 + 会话列表 + 右侧聊天区 + 底栏输入区等），使用 **Qt Widgets + `.ui` + QSS** 实现；**不使用 QML**。参考图可在实现前收集截图或官方/公开界面示意，仅作**布局与交互参考**，避免直接拷贝受版权保护的美术素材与图标，资源尽量使用**自绘、开源图标或授权素材**。

---

## UI 设计开发（仿 QQ 风格，贯穿各阶段）

在功能开发的同时推进界面，避免最后阶段才补皮。建议顺序如下。

### 1. 参考与信息架构（可与阶段 0～1 并行）

- 整理**参考图**（自备截图或示意图）：登录/注册页、主窗口三栏（或两栏）结构、聊天窗口气泡与输入区、好友列表行样式等；**PNG 统一放在仓库根目录 `ui参考图/`，并在 [UI参考.md](./UI参考.md) 的索引表中登记文件名与说明。**  
- 用一页纸或简单草图定**信息架构**：登录态 → 主窗口（会话 / 联系人 / 设置入口）→ 单聊窗口（内嵌或堆叠在同一主窗内，与 Qt `QStackedWidget` / `QSplitter` 等选型一致即可）。  
- 在 `qt-client/resources/` 规划图标、字体与色板（主色、背景、分割线、选中态），与 `src/style/` 中 QSS 或样式封装对应。

### 2. 视觉与组件分层（与阶段 2 起衔接）

- **全局样式**：在 `src/style/` 集中维护 QSS 片段或按窗口注入样式表，避免在业务 `.cpp` 里散落长字符串。  
- **复用控件**：会话列表行、好友行、头像圆角、时间戳副标题等，可逐步抽成独立 `QWidget` 子类放在 `src/ui/`，便于统一仿 QQ 的密度与对齐。  
- **主窗口骨架**：尽早做出**接近 QQ 的框架**（即使后端仍为空）：左侧窄条图标、中间列表、右侧空白聊天区，方便后续只换数据不接布局。

### 3. 各功能对应的界面重点（与下列阶段条目对应）

| 功能阶段 | UI 侧重点（仿 QQ 习惯） |
|----------|-------------------------|
| 注册 / 登录 | 简洁表单、主色按钮、错误提示区域；验证码与倒计时按钮（阶段 3） |
| 好友 | 列表行（头像 + 昵称 + 在线角标/颜色）、搜索框置顶、申请列表抽屉或子页 |
| 消息 | 会话列表高亮选中、未读数角标、聊天气泡左右对齐、底部工具栏（表情/文件等可先占位） |
| 文件 | 传输进度条、与聊天区协调的弹窗或内嵌进度 |
| 打磨 | 托盘图标、窗口最小化行为、字体与 DPI 粗略适配、主要窗口截图用于论文 |

### 4. 验收（UI 相关）

除各阶段功能验收外，答辩前宜满足：**主流程界面风格统一**、**三区块布局清晰可辨**（用户能直观看出“像常见 PC 聊天软件”）、无明显错位与遮挡；论文插图可使用**本客户端实拍截图**（脱敏数据）。

## 阶段 0：环境与骨架（已完成目标）

- [x] 仓库分为 `qt-client`、`vs-server`，各自独立构建。  
- [x] 客户端目录：`app` / `ui` / `style` / `utils` / `resources`。  
- [x] 服务端：`src/main` + `include/vsserver` 占位，预留 `net` / `db`。

**验收**：两端空工程本地均能编译运行。

---

## 阶段 1：TCP 连通与协议壳

**目标**：客户端与服务端建立连接，能收发**最小封包**（如心跳、版本握手），并解决粘包/半包。

- 服务端：监听、accept、每连接读写循环或异步模型。  
- 客户端：`QTcpSocket` 封装，信号槽对接 UI 状态（已连接/断开）。  
- 约定：`magic` + `protocol_version`（与头文件一致）。  
- **UI（可选并行）**：主窗口**空壳布局**（仿 QQ 分栏：左栏占位 + 会话列表占位 + 右侧内容区），标题栏/状态栏可先简化；用于后续阶段直接填业务控件。

**验收**：两端互相能连续发送 1000 条以上测试包无错位；断网重连后状态正确。

---

## 阶段 2：SQLite 与用户注册 / 登录闭环

**目标**：落库用户表；密码加盐哈希；登录后维持会话（session / token 选型实现时确定）。

- 建表与 DAO 层（对齐论文 4.3）。  
- 协议：`auth_register`、`auth_login`、`error` 错误码。  
- 客户端：**登录 / 注册页**与网络层对接；**仿 QQ 的简洁登录视觉**（背景区、主按钮、输入框层级）；注册与登录同一栈切换或分页，与参考图一致即可。

**验收**：新用户注册后可登录；错误密码拒绝；数据库内无明文密码。

---

## 阶段 3：邮箱验证码与邮件辅助服务

**目标**：`req_email_code` → 主服务端生成验证码 → 调辅助服务发信 → 注册时校验。

- 主服务端：频率限制、过期、尝试次数。  
- 辅助服务：最小 HTTP 接口 + 密钥校验 + SMTP 配置外置。  
- 客户端：验证码输入、**倒计时与禁用重发**（按钮态与 QQ 类应用一致）；错误提示文案区域不挤压主布局。

**验收**：离线环境下可用测试邮箱或抓包验证链路；主服务端日志可追溯请求（脱敏）。

---

## 阶段 4：好友模块

**目标**：搜索用户、发起申请、列表、同意/拒绝、删除好友；在线通知（若已具备推送通道）。

- 协议：`friend_*` 系列。  
- 客户端：**联系人 / 好友列表**（参考 QQ：左栏入口、列表行头像+昵称+在线状态、顶栏搜索）；**好友申请**列表（新申请角标或加粗，可选）；右键或菜单删除好友等交互可先简化为按钮。

**验收**：两账号互加好友后列表一致；拒绝/删除后状态与库一致。

---

## 阶段 5：即时消息

**目标**：点对点文字消息；在线转发、离线落库；基础历史拉取（分页）。

- 协议：`msg_send`、`msg_push`、`msg_ack`（粒度可选）。  
- 客户端：**会话列表**（最后一条预览、时间、未读角标，样式参考 QQ）；**聊天区**（己方/对方气泡左右分布、头像、可选时间分隔）；**输入区**（多行输入 + 发送键，Ctrl+Enter 发送等习惯可与参考图一致）；列表与聊天区**滚动**与选中态优化。

**验收**：双开客户端消息互通；一方离线后上线能收到离线消息（策略与论文一致）。

---

## 阶段 6：文件分片传输

**目标**：大文件分片、进度、校验；传输记录入库（论文 4.2.4）。

- 协议：`file_*` 系列。  
- 客户端：**发送/接收文件**入口（工具栏按钮或拖拽区，可先占位）；**进度条**样式与主窗口协调；暂停/继续（能力以实现为准）。

**验收**：局域网内传输大于例如 100MB 文件成功且哈希一致。

---

## 阶段 7：打磨与非功能

**目标**：心跳与超时、日志、配置项（服务器 IP/端口）、安装/部署说明、与论文图表对齐的截图与测试记录。

- **UI 打磨**：统一 QSS 与间距；**托盘图标**与最小化到托盘（可选）；主窗口标题与任务栏图标；高 DPI 下抽查；全窗口走查避免截断。  
- **论文素材**：截取登录、主界面、聊天、好友等界面**脱敏截图**插入论文或答辩 PPT。

**验收**：答辩演示路径通畅；论文中承诺的核心功能均有对应操作可展示。
