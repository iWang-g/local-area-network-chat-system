# 第2章 相关技术介绍

## 2.1 C++ 编程语言

C++ 是一门成熟且高效的编程语言，在系统软件开发、高性能与实时应用中应用广泛。在局域网聊天系统中，选择 C++ 作为**服务端与客户端业务层**的基础语言，主要基于以下优势：

1. **高性能与资源管控**：C++ 具备明确的资源与生命周期语义，便于在服务端对连接、缓冲与线程等资源进行精细管理；结合 **TCP/IP Socket** 接口完成收发，有利于控制开销并保障消息与文件传输的实时性。

2. **模块化与可维护性**：借助封装、继承与多态等机制，可将用户管理、会话维护、消息路由、文件传输等能力拆分为相对独立的模块，降低耦合，便于后续扩展（如群聊、富媒体等）。

3. **跨平台服务端实现**：在遵循标准语言与可移植接口的前提下，服务端逻辑可在 **Windows、Linux** 等环境下编译部署，契合本课题对服务端跨平台能力的要求。

在本系统中，**服务端**的核心能力（监听与接入、协议解析、鉴权与路由、数据库访问等）由 C++ 实现；**客户端**除界面层外，其业务控制与网络收发等同样以 C++ 为主，并与 Qt 框架协同工作。

## 2.2 Qt 6 跨平台开发框架

Qt 是由 **The Qt Company** 主导维护的跨平台 C++ 应用程序框架，广泛用于图形界面与配套基础设施开发。本课题**客户端**采用 **Qt 6**，以利用其成熟的 API 与跨平台发布能力。

1. **信号与槽机制（Signals and Slots）**：用于对象间松耦合通信。客户端中可将网络数据的到达、界面状态变化等事件以信号形式分发，由槽函数完成展示与业务联动，从而降低回调嵌套复杂度。

2. **Qt Network 模块**：提供对 TCP/UDP 等协议的封装。本系统**客户端**通过 `QTcpSocket` 等与聊天**服务端**建立 **TCP** 长连接，在应用层配合自定义协议完成消息的组包与拆包；粘包、半包等问题的处理在设计与实现章节中结合具体协议说明。

3. **跨平台界面与部署**：Qt 6 支持在 Windows、Linux 等目标平台上构建统一的客户端工程，有利于在局域网多终端环境中保持一致的用户体验。

需要说明的是：**聊天服务端**的网络监听与并发处理采用标准 **C++ Socket** 技术路线实现，与客户端所使用的 Qt Network 模块在工程上相互独立，但共同基于 **TCP/IP** 完成互通。

## 2.3 SQLite 嵌入式数据库

SQLite 是一款轻量级嵌入式关系型数据库，以单文件形式存储数据，无需独立数据库服务进程。

1. **零配置与易部署**：数据库即文件，拷贝与备份成本低，适合局域网场景下快速部署与迁移。

2. **事务与一致性**：SQLite 支持 **ACID** 事务，有利于保证用户资料、好友关系与聊天记录等在异常退出或断电场景下的数据完整性（具体并发访问策略在实现中结合连接与锁机制设计）。

3. **结构化查询能力**：基于 **SQL** 可较自然地实现聊天记录分页、条件筛选与统计等需求。

在本系统中，**服务端**使用 SQLite 持久化用户、好友关系与消息等核心业务数据；**客户端**以界面与网络交互为主，是否引入本地数据库用于缓存或辅助展示，可在实现阶段按需求取舍，本章从架构上保留该扩展空间。

## 2.4 TCP/IP 协议与 Socket 通信

局域网聊天系统的通信基础是 **TCP/IP** 协议族与 **Socket** 编程接口。

1. **TCP 的可靠性**：即时消息与文件分片对完整性与顺序性要求较高，本系统业务连接采用 **TCP**。其连接建立、重传、流量控制等机制为上层应用提供了有序、可靠的字节流抽象。

2. **Socket 接口与 C/S 模型**：**Socket** 是操作系统提供的网络编程接口；在工程表述中常统称为 **TCP/IP Socket** 编程。服务器端监听端口并接受连接，客户端发起连接后，双方在已建立的连接上进行全双工读写，由应用层协议定义消息边界与语义。

本系统的**服务端**直接基于各平台的 Socket API（如 Windows 下的 **Winsock** 与 Linux 下的 **Berkeley Socket**）完成网络层能力；**客户端**在 Qt 中对 Socket 能力进行封装调用，二者在协议层面保持一致即可互通。

## 2.5 多线程编程技术

多用户接入与耗时操作并存时，多线程有助于提升吞吐并改善交互体验。

1. **客户端：界面与耗时任务分离**：文件读写、大批量历史记录加载等操作若阻塞主线程易导致界面“假死”。将耗时工作置于工作线程，通过 Qt 推荐的跨线程通信方式回传结果，可保持界面响应。

2. **服务端：并发连接处理**：可采用“每连接一线程”、线程池或 **I/O 多路复用** 等策略（具体选型在设计与实现章节给出），以同时维护多路 **Socket** 连接，完成收包、鉴权、转发与落库等处理。

## 2.6 邮箱验证码与 SMTP

注册与登录环节采用**邮箱验证码**时，通常需要向用户邮箱发送一次性验证码。主流邮箱服务商（如 **QQ 邮箱、163 邮箱**）提供 **SMTP** 发信能力，并常要求使用**授权码**而非登录密码完成身份校验。

从系统结构上看，可将能力拆分为两部分：**主服务端**负责验证码生成、有效期、次数限制与校验结果写入等业务逻辑；**邮件投递**则可由体量较小的**独立辅助服务**承担，通过 **SMTP** 连接邮件服务器完成发送。这样可降低主服务端在邮件协议与加密传输方面的实现与维护成本，也便于在部署层单独配置邮箱账号与发信参数（如服务器地址、端口、TLS 方式等）。

---

通过上述技术的组合：**客户端**以 **Qt 6** 实现人机交互与 **TCP** 长连接接入，**服务端**以 **C++** 与 **TCP/IP Socket、多线程** 等实现高并发处理与数据持久化（**SQLite**），并在需要时辅以 **SMTP** 发信链路完成邮箱验证，从而构成完整的局域网即时通讯技术方案。
